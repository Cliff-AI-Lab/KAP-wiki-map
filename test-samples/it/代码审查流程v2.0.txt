代码审查流程 V2.0
文档编号：DEV-2025-003
发布日期：2025年9月1日
归属部门：研发效能部
审批人：陈浩然（研发总监）

一、目的与范围

本流程规范了公司所有研发团队的代码审查（Code Review）活动，覆盖从 Pull Request 提交到代码合入主干的完整流程。适用于所有 Git 仓库（含 GitLab 自建实例和 GitHub Enterprise）。

二、PR 提交规范

1. 分支命名：feature/JIRA编号-简要描述（如 feature/PAY-1234-add-refund-api）
2. 单次 PR 变更行数建议不超过 400 行（不含自动生成代码和测试用例）。超过 800 行的 PR 须拆分，否则审查人有权拒绝审查。
3. PR 描述必须包含：
   - 关联的 JIRA 工单号
   - 变更摘要（做了什么、为什么做）
   - 影响范围评估（涉及哪些模块/接口）
   - 自测结果截图或日志
4. PR 模板已内置于各仓库 .gitlab/merge_request_templates/default.md。

三、审查人分配规则

1. 每个 PR 至少需要 2 名审查人，其中 1 名须为该模块的 Code Owner。
2. Code Owner 清单维护在仓库根目录 CODEOWNERS 文件中，按季度更新。
3. 审查人须在 PR 提交后 4 个工作小时内开始审查，24 小时内完成首轮意见。
4. 跨团队变更须额外增加对方团队的 Tech Lead 作为审查人。

四、审查检查清单

审查人须逐项确认以下内容：

【功能正确性】
- [ ] 业务逻辑是否符合需求文档描述
- [ ] 边界条件和异常场景是否处理
- [ ] 并发安全性是否考虑（锁、事务隔离级别）

【代码质量】
- [ ] 命名是否清晰、符合团队编码规范
- [ ] 是否存在重复代码可提取为公共方法
- [ ] 圈复杂度是否超过 15（SonarQube 自动检测）

【安全合规】
- [ ] 是否存在 SQL 注入、XSS 等安全风险
- [ ] 敏感信息（密钥、Token）是否通过配置中心管理
- [ ] 日志中是否脱敏处理用户隐私数据

【测试覆盖】
- [ ] 新增代码单元测试覆盖率不低于 80%
- [ ] 关键路径是否有集成测试
- [ ] CI 流水线是否全部通过（lint、test、build）

五、审批与合入规则

1. 所有检查项通过 + 2 名审查人 Approve = 可合入。
2. 合入方式统一采用 Squash Merge，保持主干历史整洁。
3. 合入后自动触发 CI/CD 流水线部署至 Dev 环境。
4. 紧急热修复（Hotfix）可由 Tech Lead 单人审批后合入，但须在 24 小时内补充完整审查。

六、度量指标

研发效能部按月统计以下指标并通报各团队：
- PR 平均审查时长（目标 < 8h）
- 首次审查通过率（目标 > 60%）
- 千行代码缺陷密度（目标 < 2.5）

生效日期：2025年9月1日起全公司执行。
