微服务架构设计规范 V3.2
文档编号：ARCH-2024-007
发布日期：2025年11月15日
归属部门：基础架构部
审批人：李明远（首席架构师）

一、总则

本规范适用于公司所有新建及重构的微服务项目，旨在统一技术选型、服务治理和部署标准。所有业务线在启动微服务拆分前，须经架构评审委员会审批通过。

二、API 网关规范

1. 统一采用 Kong Gateway 3.x 作为 API 网关，禁止各团队自建网关组件。
2. 所有对外暴露的接口必须通过网关统一路由，网关层负责限流（默认 QPS 上限 500/服务）、鉴权和日志采集。
3. 接口响应时间 P99 不得超过 800ms，超时阈值设置为 3000ms。
4. API 版本管理采用 URL 路径方式（如 /api/v1/），废弃版本须保留至少 6 个月兼容期。

三、服务网格（Service Mesh）

1. 生产环境统一部署 Istio 1.20，Sidecar 注入策略为 namespace 级别自动注入。
2. 服务间通信采用 mTLS 加密，证书由内部 CA 签发，有效期 90 天自动轮转。
3. 流量治理规则：
   - 灰度发布：基于 Header（x-canary: true）路由，金丝雀权重不超过 10%。
   - 熔断策略：连续 5xx 错误达 50% 时触发熔断，冷却期 30 秒。
   - 重试机制：仅对 GET 请求启用自动重试，最多 2 次。

四、容器化部署标准

1. 所有服务必须提供 Dockerfile，基础镜像统一使用公司内部 Harbor 仓库（harbor.internal.com）。
2. 推荐基础镜像：
   - Java 服务：eclipse-temurin:21-jre-alpine
   - Go 服务：golang:1.22-alpine（构建）+ scratch（运行）
   - Python 服务：python:3.12-slim
3. 容器资源配额：
   - 核心服务：CPU 请求 500m / 限制 2000m，内存请求 512Mi / 限制 2Gi
   - 非核心服务：CPU 请求 200m / 限制 1000m，内存请求 256Mi / 限制 1Gi
4. 健康检查必须配置 liveness 和 readiness 探针，就绪探针初始延迟不超过 30 秒。
5. Kubernetes 命名空间按业务线划分：biz-order、biz-payment、biz-user、platform-common。

五、日志与可观测性

1. 日志格式统一为 JSON，必须包含 traceId、spanId、timestamp、level、service 字段。
2. 链路追踪接入 Jaeger，采样率生产环境设为 1%，预发布环境设为 100%。
3. 核心指标通过 Prometheus + Grafana 监控，各服务须暴露 /metrics 端点。
