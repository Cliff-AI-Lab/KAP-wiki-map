## 0. 文档说明

### 0.1 与技术决策书的关系

PRD 是技术决策书的产品化展开。技术决策书回答"建什么 / 怎么建"，PRD 回答"用户体验什么 / 验收标准是什么"。

**约定**：本 PRD 不重复决策书已写明的架构细节，仅引用决策编号（如 D8、D13）。修改架构请回到决策书走变更评审。

### 0.2 文档约定

- **功能编号**：`F<块>.<模块>.<序号>`，如 `F1.3.2` 表示块① 第 3 模块第 2 个功能
- **优先级**：MoSCoW 法
  - **M (Must)**：MVP 必做
  - **S (Should)**：MVP 后第一版本应做
  - **C (Could)**：可选，看资源
  - **W (Won't)**：本期不做
- **用户故事格式**：作为 [角色]，我希望 [行为]，以便 [价值]
- **验收标准格式**：可观察、可测量的条件清单

## 1. 产品概述

### 1.1 业务背景

企业（首批锁定制造与能源，向全行业扩展）普遍存在以下知识管理痛点：

| 痛点 | 现状 | KAP 应对 |
|---|---|---|
| 知识体系混乱 | 文档散落多套系统（OA / 共享盘 / 个人电脑 / 邮件），无统一分类 | 块① 智能体引导建立 4 级主树 + Facet |
| 入库即失联 | 文档入库后无人维护，标准升版 / 制度更替没人跟 | 块② 治理 Agent 持续审计 + 时效性预警 |
| RAG 答非所问 | 直接接入 RAG，幻觉严重，无溯源 | 块③ 三路并行 + Wiki 缓存 + 强溯源 |
| 涉密文档不能用 | 核心研发资料不敢入库，怕泄露 | 敏感实体识别 + 双层存储 + 密级路由 |
| 业务专家被"工具化" | 让 SME 自己整理知识体系，无人响应 | AI 主导提议 + 专家审核兜底，反向解放 SME |
| 治理无量化 | 知识库覆盖度、命中率、缺口、冷门——都说不清 | W6 仪表盘 + AIOps 角色专门跟指标 |

### 1.2 产品愿景

**让企业知识从"杂乱的文档堆"变成"持续治理、可被信任、按需调用的智能体资产"。**

### 1.3 价值主张（与普通 RAG 的核心区别）

| 维度 | 普通 RAG / SaaS 知识库 | KAP |
|---|---|---|
| 体系 | 用户自建 / 平铺无层级 | AI 引导建 4 级 + Facet |
| 入库 | 简单分块向量化 | 6 工位流水线 + 4 角色审核 |
| 治理 | 一次入库，长期不动 | 5 治理 Agent + LLM-Critic 6 维质疑持续治理 |
| 消费 | 单路 RAG | Wiki + RAG + 图谱 三路并行 |
| 安全 | 文件级权限 | 行级数据权限 + 密级路由 + 实体级脱敏 |
| 闭环 | 无反馈机制 | 高频回流 + 缺口识别 + 本体演化 |
| 可信 | 黑盒 | 全链路审计 + 决策追溯 + 溯源到原文 |

### 1.4 三块产品形态总览

参见技术决策书 §1.2。本 PRD 第 3-5 章按三块分别展开功能需求。

### 1.5 范围声明

#### 在范围内（In Scope）

- 文档类知识（PDF / Word / Excel / PPT / 图 / 音 / 视）的入库、治理、消费
- 全行业的 4 级知识体系建设（首批制造与能源，通过可插拔模板包扩展至金融、政企、医疗、运营商等）
- 4 角色协同治理（DG / SME / SEC / AIOps）
- 终端用户问答与知识浏览（Reader）
- 私有化部署与运维

#### 不在范围内（Out of Scope）

- 个人 / 小团队级 Wiki（飞书文档 / 钉钉 / Notion 已覆盖）
- 实时协作编辑（如 Google Docs 多人同编）
- 工业实时数据（DCS / SCADA）的知识化（属于 IoT 平台范畴）
- 知识付费 / 内容商业化
- 公有云 SaaS

## 2. 用户与场景

### 2.1 用户画像

| 角色 | 画像描述 | 主要诉求 | 典型操作频次 |
|---|---|---|---|
| **DG 数据治理员** | 30-40 岁，IT/数据部门，懂业务也懂技术 | 知识体系建设、命名规范统一、版本管理 | 高（建体系阶段每日，运营期每周） |
| **SME 业务专家** | 35-50 岁，资深业务专家或工程师，时间紧 | 高效审核 LLM 抽取的实体关系，关心专业准确性 | 中（每日 30 分钟-2 小时） |
| **SEC 安全审计员** | 35-45 岁，合规 / 安全岗，对密级敏感 | 敏感数据不泄露、审计可查、合规可证 | 低（按需，每周抽查） |
| **AIOps AI 运营员** | 25-35 岁，AI / 数据科学背景，关心模型效果 | 监控召回率、命中率、模型版本对比、Prompt 优化 | 高（每日仪表盘） |
| **Reader 业务用户** | 20-55 岁，一线员工到管理层，对工具不熟 | 快速找到准确答案，不愿写复杂查询 | 高（每周多次到每日） |

### 2.2 典型场景

#### 场景 S1：从零搭建企业知识体系（DG 主导）

> 制造业客户首次部署 KAP，IT 部门张工（DG 角色）需在 2 周内完成 5 个事业部、12 个部门的知识体系搭建。

**步骤**：
1. 张工创建项目，选择"装备制造"行业模板
2. 上传 200 份代表性文档（覆盖各部门）
3. 块① Architect Agent 识别为"装备制造-离散"，提议 6 个一级业务域
4. 张工与各事业部副总开 1 小时对齐会，对话式调整主树到 4 级
5. Agent 提议 7 个 Facet 维度（含密级、设备维度、标准引用）
6. Agent 生成命名规范，张工审核通过
7. Agent 检测到 12 处归类冲突，张工逐个决策（保留 / 拆分 / 合并）
8. 导出体系方案，进入块② 入库阶段

**关键体验诉求**：
- AI 引导明确，不让用户面对空白画板
- 每一步可回退、可修改
- 每个决策有"AI 建议 + 用户选择"的双轨

#### 场景 S2：核心研发知识入库（SME 主审）

> 能源装备客户研发部李工（SME），每周需审核 W4 工位 LLM 抽取的实体关系约 100 条。

**步骤**：
1. 李工登录 → 自动跳转到"我的 SME 审核台"
2. 看到工单池：W4 工位 87 待审、W5 工位 12 实体合并提议
3. 点 W4 工位 → 看到批量列表（按文档分组）
4. 选第一份《汽轮机叶片振动故障处理 SOP v3.2》
5. 左侧原文，中间 LLM 抽取的 6 实体 4 关系（带高亮 + 置信度），右侧 LLM-Critic 2 处质疑
6. Critic 提示："实体[振动分析工程师]在 KB-DEV-0091 中定义为[设备诊断岗]，疑似同义"
7. 李工点[合并]，选择保留[设备诊断岗]，给出理由"业务部新岗位名"
8. LLM 推荐"将所有[振动分析工程师]迁移为[设备诊断岗]"，李工确认
9. 系统全局更新（所有相关 chunk + 图谱节点），写审计日志
10. 工单标 approved，李工进入下一份

**关键体验诉求**：
- 审核不打断节奏：批量、键盘快捷、判例复用
- LLM 提议明确建议，李工只做"通过 / 修改 / 驳回"
- 决策即生效，但有 7 天回滚窗口

#### 场景 S3：终端业务员一句话提问（Reader）

> 现场操作工小王半夜值班发现汽轮机异常，手机问 KAP："汽轮机1#振动故障应急处置流程是什么？"

**步骤**：
1. 小王打开消费门户，输入查询
2. 三路并行同时跑：Wiki 命中"动火/振动应急处置 v2.1"（置信 0.94）→ 短路返回
3. 200ms 内出答案：分级流程 + 操作步骤 + 联系人
4. 答案带"路由徽章 = wiki 短路"+ "来源 3 条"，小王展开看到原文 SOP 第 3.2 节
5. 小王点[← SOP 原文]→ 弹原文 popup，确认无误
6. 小王点[这条答案有用]→ 反馈记录，强化此 Wiki 条目权重

**关键体验诉求**：
- 5 秒内出答案（含网络）
- 答案必须带溯源
- 移动端友好

#### 场景 S4：本体演化 SME 审批（SME + DG）

> 季度全量重抽前夜，AIOps 老赵收到提醒："过去 3 个月累计未匹配实体 78 个，建议演化本体"。

**步骤**：
1. 老赵进入治理工作台 → 本体管理 → 演化提议
2. 看到 LLM 提议清单：新增 5 实体类型、3 关系类型，拆分 1 既有关系
3. 每条提议带：依据样本 + 影响范围预估 + 历史相似演化记录
4. 老赵把提议转给 SME 委员会（5 人）审批
5. SME 委员会在线投票，3/5 通过新增"工艺缺陷"实体类型
6. 系统启动影子图谱重抽，预估 8 小时完成
7. 重抽完成，对比报告：实体覆盖率 +12%、关系数量 +234、检索命中率 +5%
8. SME 老李审批通过 → 灰度切换到主图谱
9. 7 天观察期开始，AIOps 持续监控

**关键体验诉求**：
- 演化全流程可视化
- 影响预估清晰，不让 SME 盲投
- 回滚一键可达

## 3. 块① 知识咨询智能体（建体系）· 功能需求

### 3.1 模块概述

块① 是一个**对话式 AI 顾问产品**，引导客户从无到有建立 4 级知识体系 + Facet + L2 本体草案 + 命名规范。

入口：`/agent/architect`

### 3.2 功能清单

#### F1.1 项目初始化与材料上传 [M]

**功能描述**：用户创建知识体系项目，上传代表性材料用于行业识别。

**子功能**：
- F1.1.1 创建项目，填写基本信息（名称、行业初判、负责人）[M]
- F1.1.2 拖拽上传或目录上传文档（支持批量、断点续传）[M]
- F1.1.3 实时显示上传进度、解析结果、识别格式 [M]
- F1.1.4 文档预览（前 10 页可视）[S]
- F1.1.5 上传材料数量建议（最少 50 份，推荐 200+）[M]
- F1.1.6 支持从已有共享盘 / SharePoint / OneDrive 拉取（连接器）[C]

**用户故事**：
- 作为 DG，我希望能批量上传几百份文档，以便 AI 有足够样本识别行业。
- 作为 DG，我希望上传后立即看到解析成功 / 失败统计，以便快速决定是否补充材料。

**验收标准**：
- [ ] 单批次支持 1000 份文档上传
- [ ] 单文档大小限制 100MB（可配置）
- [ ] 解析失败的文档明确给出原因
- [ ] 上传中断后可恢复

#### F1.2 行业识别与模板推荐 [M]

**功能描述**：AI 根据上传材料自动识别行业及子方向，推荐对应行业模板包。

**子功能**：
- F1.2.1 行业自动识别（基于关键词 + 文档类型 + LLM 推断）[M]
- F1.2.2 子方向识别（如制造 → 离散制造 / 流程制造 / 装备制造）[M]
- F1.2.3 行业模板推荐（含模板预览：主树骨架、Facet、本体）[M]
- F1.2.4 用户可手动调整识别结果（覆盖 AI 判断）[M]
- F1.2.5 识别置信度展示（高 / 中 / 低 + 依据样本）[S]
- F1.2.6 支持多模板融合（如制造 + 能源装备）[M]

**用户故事**：
- 作为 DG，我希望 AI 准确识别我的行业，以便不需要从空白模板开始。
- 作为 DG，当 AI 识别错时，我希望能快速纠正而不是重来。

**验收标准**：
- [ ] 制造 / 能源主行业识别准确率 ≥ 90%（基于内部测试集）
- [ ] 识别耗时 < 30 秒（200 份样本）
- [ ] 模板推荐至少 1 个、最多 3 个
- [ ] 用户可手动指定不在推荐列表中的行业模板

#### F1.3 主树对话式构建 [M]

**功能描述**：AI 引导用户对话式构建 4 级主树（公司-部门-岗位-操作），支持增删改、合并、拆分、重排。

**子功能**：
- F1.3.1 AI 提议初版主树（基于行业模板）[M]
- F1.3.2 树形可视化（左右分屏：左对话 / 右树）[M]
- F1.3.3 节点 CRUD（添加 / 重命名 / 删除 / 移动）[M]
- F1.3.4 节点合并 / 拆分 [M]
- F1.3.5 文档自动归类到节点（基于上传材料）[M]
- F1.3.6 节点级别检查（强制 4 级，但允许局部少于 4 级）[M]
- F1.3.7 历史版本（撤销 / 重做 / 版本快照）[S]
- F1.3.8 多人协同（DG + 业务负责人同时编辑）[C]
- F1.3.9 主树导入导出（YAML / JSON）[M]

**用户故事**：
- 作为 DG，我希望与 AI 对话调整主树，比如说"把'设备维护'拆成'预防性维护'和'故障维护'"，AI 立即在树上反映。
- 作为 DG，我希望看到每个节点关联的文档数，以便判断是否合理。

**验收标准**：
- [ ] 节点数支持 1-1000 范围
- [ ] 拖拽移动节点 < 100ms 响应
- [ ] AI 调整建议明确（如"建议把 X 移到 Y 下，因为...")
- [ ] 调整不丢失已关联的文档

#### F1.4 Facet 多面分类配置 [M]

**功能描述**：AI 提议 Facet 维度，用户配置取值字典。

**子功能**：
- F1.4.1 AI 提议 Facet 维度（基于行业模板 + 上传材料）[M]
- F1.4.2 维度增删改 [M]
- F1.4.3 维度取值字典维护（如密级：公开/内部/秘密/机密）[M]
- F1.4.4 维度优先级排序（影响检索时的过滤面板顺序）[S]
- F1.4.5 维度互斥规则配置（如"长期有效"与"一次性"互斥）[C]
- F1.4.6 维度模板复用（跨项目复用 Facet 配置）[S]

**验收标准**：
- [ ] 至少支持 3-15 个 Facet 维度
- [ ] 每个维度的取值字典可独立维护
- [ ] AI 提议必须基于上传材料，不能凭空（要带样本依据）

#### F1.5 命名规范生成 [M]

**功能描述**：基于主树 + Facet，AI 生成命名规范，DG 审核确认。

**子功能**：
- F1.5.1 AI 提议命名规范（拼接顺序 / 分隔符 / 字段必填性）[M]
- F1.5.2 实时预览（用真实样例展示命名效果）[M]
- F1.5.3 规范导出（含校验脚本）[M]
- F1.5.4 命名违规检测（已上传文档批量校验）[S]
- F1.5.5 历史命名规范变更日志 [S]

**默认规范模板**（参考决策书 §4.4）：
```
[层级编码]-[业务域代码]-[文档类型]-[标题]-[版本]-[密级]-[Owner]-[生命周期态]
示例：KB-CS-SOP-投诉处理-v2.3-内部-客服部-生效中
```

**验收标准**：
- [ ] 默认规范开箱即用，但可调整字段顺序与必填性
- [ ] 规范变更必须留版本，旧文档不强制重命名（标"旧规范"标识）

#### F1.6 冲突检测与归类预演 [M]

**功能描述**：AI 在体系完成前预演归类，检测潜在冲突 / 重复 / 无处归类。

**子功能**：
- F1.6.1 一键预演（用上传的所有材料尝试归类）[M]
- F1.6.2 冲突清单（同一文档落入 2+ 节点）[M]
- F1.6.3 重复清单（高相似度文档对）[M]

## 11. 风险与开放问题

### 11.1 产品风险

| 风险 | 影响 | 应对 |
|---|---|---|
| SME 不愿意用 | 全盘失败 | 块① 用 AI 顾问降低门槛；块② 审核台 LLM 预填让 SME 像审稿人不像录入员 |
| 客户行业模板不在预置范围 | 块① 体验差 | 提供"通用模板 + 客户共建"路径 |
| 私有化部署算力紧张 | 性能不达标 | 量化部署 + 异步队列 + 模型预热；最小 / 标准 / 大型三档资源画像 |
| 多语言文档（中英混合） | 抽取召回率低 | 嵌入模型选支持中英的（bge-large-zh）+ Refiner Prompt 多语言适配 |
| 用户期望"问答全自动" | 低估治理工作量 | 销售前期就讲清"AI-native + 人工兜底"产品定位 |

### 11.1.1 复用相关风险（v1.1 新增）

| 风险 | 影响 | 应对 |
|---|---|---|
| Wiki-map V15 同步 LLM 阻塞坑被带入 KAP | M0 演示期 API 挂起 | M0 启动第一周必须切 AsyncClient（决策书 §13） |
| Wiki-map V15 mock embedding 残留 | 演示效果差，召回精度低 | M0 起就用真嵌入（bge-large-zh），mock 仅做单测 |
| Wiki-map V15 内存 GraphStore 重启丢失 | 客户演示翻车 | M0 末期就完成 Neo4j 持久化迁移 |
| ISS 微服务粒度过细，私有化算力紧张 | 资源浪费 | 启用 ISS 双模式部署（API/Parser 合并），最小化合规副作用 |
| Wiki-map V15 与 ISS 风格混搭 | 审美不统一 | M4 UI 重设计统一为工程蓝图美学（决策书 §10.1）|
| 团队不熟悉两套技术栈 | 复用变重写 | M0 启动前组织 Wiki-map V15 + ISS 代码走读 1-2 天 |
| Wiki-map V15 没有制造行业模板 | M1 卡壳 | M0 期间业务侧并行准备制造模板（无须开发资源）|
| 复用代码缺乏测试覆盖 | 改造引入 bug 风险高 | 复用前先用现有金标 QA（Wiki-map V15 energy-test-results）回归 |
