# KAP 知识智能体平台 · 三中心部署资源清单

> 版本：v1.0.0a0（M21）
> 编制日期：2026-05-05
> 适用范围：私有化部署（制造业 / 能源行业等企业知识库底座）

---

## 一、平台架构总览

KAP 平台按业务边界拆为三个**松耦合可独立部署**的中心，每个中心可单独使用，也可通过 HTTP API 串联无缝衔接。

| 中心 | 业务定位 | 工位范围 | 默认端口 |
|:---|:---|:---|:---:|
| **咨询中心 (architect)** | AI 知识架构师对话 + 完整知识体系建立流程 | W1 项目 → W2 上传 → W3 去噪审核 → W4 体系 → W5 Wiki 编织 | 8011 |
| **知识中心 (storage)** | 存储 / 向量化 / 图谱化 / 治理审核 | W6 入库 + 向量化 + 图谱化 + 4×6 矩阵审核 + 影子库重抽 | 8012 |
| **消费中心 (portal)** | 渐进式三路召回 + 运营仪表盘 | Wiki 快路径 / RAG 深检索 / 图谱跨域 三路并行 | 8013 |

**一体化部署**额外提供单镜像方案，端口 8001。

### A.3 共享基础设施清单

下表中**全部基础设施由三中心共享**，单独物理实例或集群均可。

| 组件 | 用途 | 谁用 | 推荐资源 | 镜像 |
|:---|:---|:---|:---:|:---|
| PostgreSQL 16 | 元数据 / 决策日志 / 查询日志 / 召回评估 / Wiki 质量 / 抽取诊断 / 项目 / 文档 / 本体版本 | 全 | 4 vCPU / 8 GB / 200 GB SSD | `postgres:16-alpine` |
| Neo4j 5 Community | 知识图谱（实体 / 关系 / 影子库） | 知识中心 / 消费中心 | 4 vCPU / 8 GB / 100 GB SSD（生产推荐独立物理实例） | `neo4j:5-community` + APOC |
| Milvus 2.4 | 向量索引（chunk embeddings） | 知识中心 / 消费中心 | 4 vCPU / 8 GB / 100 GB SSD | `milvusdb/milvus:v2.4.17` |
| Redis 7 | 缓存 / 会话 / ISS LoginUser 拉取 | 咨询中心 / 消费中心 / 知识中心 | 1 vCPU / 2 GB | `redis:7-alpine` |
| MinIO | 文档原件归档 | 咨询中心 / 知识中心 | 2 vCPU / 4 GB / 500 GB | `minio/minio:latest` |
| etcd v3.5 | Milvus 元数据 | Milvus 内部 | 1 vCPU / 1 GB | `quay.io/coreos/etcd:v3.5.16` |

| 咨询中心 | `KAP_WIKI_QUALITY_PG=1` | `wiki_quality_scores` | Wiki 6 维质量评分（M17+M19） |

| 消费中心 | `KAP_QUERY_LOG_PG=1` | `query_events` | 查询召回埋点 + 用户反馈（M7+M8） |
| 消费中心 | `KAP_RECALL_EVAL_PG=1` | `recall_eval_*` | 召回评估 + 趋势告警（M9） |

### A.5 服务间通讯（无缝衔接）

服务间走 HTTP API，由 `packages/integration/clients.py` 提供 SDK：

| 调用方 → 被调用方 | 用途 | 环境变量 |
|:---|:---|:---|
| 咨询中心 → 知识中心 | 落 L2 本体 / Wiki 入库 | `KAP_STORAGE_BASE=http://kap-storage:8012` |
| 消费中心 → 知识中心 | 拉 Wiki / 文档 / 图谱 | `KAP_STORAGE_BASE=http://kap-storage:8012` |
| 任意中心 ↔ 任意中心 | 健康检查 / 元数据 | `KAP_INTEGRATION_TOKEN=<bearer>` |

### A.6 通用环境变量

| 变量 | 必填 | 说明 |
|:---|:---:|:---|
| `KAP_ENV` | ✓ | dev / sandbox / prod |
| `POSTGRES_HOST` / `POSTGRES_PORT` / `POSTGRES_USER` / `POSTGRES_PASSWORD` | ✓ | PG 连接 |
| `NEO4J_URI` / `NEO4J_USER` / `NEO4J_PASSWORD` | ⚠ | 知识/消费中心需要 |
| `MILVUS_HOST` / `MILVUS_PORT` | ⚠ | 知识/消费中心需要 |
| `REDIS_URL` | ⚠ | 全中心可选 |
| `MINIO_ENDPOINT` / `MINIO_ACCESS_KEY` / `MINIO_SECRET_KEY` | ⚠ | 咨询/知识中心需要 |
| `IRUIDONG_API_KEY` / `IRUIDONG_BASE_URL` | ✓ | 睿动 LLM 网关（决策书 §13） |
| `JWT_SECRET` | ✓ | API JWT 签名（≥ 64 字符） |
| `KAP_AUTH_MODE` | ✓ | api_key / jwt / gateway_header（sandbox/prod 强制非 api_key） |
| `ISS_JWT_SECRET` | ⚠ | jwt 模式必填（HS512 共享密钥） |
| `CORS_ORIGINS` | ✓ | 允许跨域来源（CSV） |

### A.7 部署命令（单中心独立）

```bash
# 仅启动咨询中心
docker compose -f infra/docker-compose.split.yml --env-file .env.prod up kap-architect

# 启动全部三中心 + 共享基础设施 + nginx 网关
docker compose -f infra/docker-compose.split.yml --env-file .env.prod up

# 验证（独立健康检查）
curl https://kap.local/health/architect
curl https://kap.local/health/storage
curl https://kap.local/health/portal
```

### A.8 nginx 网关路由分发

| 路径 | 转发到 |
|:---|:---|
| `/api/v1/architect/*` | kap-architect:8011 |
| `/api/v1/{ontology,governance,wiki,projects,knowledge,sensitive,rebuild,audit,analysis}/*` | kap-storage:8012（governance/rebuild/audit 实在 storage；其余在 architect — 实际按当前 app_factory 分配） |
| `/api/v1/{qa,observability,recall-test,iss-job}/*` | kap-portal:8013 |
| `/health/{architect,storage,portal}` | 各中心 `/health` |
| 其他 | 默认到知识中心 SPA |

> 实施提示：nginx-split.conf 已配置好分发规则，按需调整后启动。

### A.9 资源汇总（单一客户机房）

| 类型 | 数量 | 总 CPU | 总内存 | 总磁盘 |
|:---|:---:|:---:|:---:|:---:|

## 四、两种方案对比

| 维度 | A 三中心分别部署 | B 一体打包部署 |
|:---|:---|:---|
| 适用场景 | 生产 / 高 QPS / 大企业 | PoC / 演示 / 小客户 / 二次开发 |
| 单点故障 | 任一中心挂不影响另两个 | 全停 |
| 资源 | 48 vCPU / 92 GB | 21 vCPU / 41 GB |
| 升级 | 可灰度（仅升咨询不动消费） | 全量重启 |
| 镜像数 | 4（含 nginx） | 1-2 |
| 网络 | 内部 HTTP 互调（StorageClient） | 进程内函数调用 |
| 运维复杂度 | 中 | 低 |
| 横向扩展 | 各中心独立扩 | 整体扩 |

## 五、共用部署前置清单

### 5.1 客户机房

- [ ] 操作系统 CentOS 7+ / Ubuntu 22.04 / RHEL 8+
- [ ] Docker 24+ 与 Compose v2
- [ ] 域名解析到客户内网 IP（如 `kap.local`）

### 5.3 ISS 集成（如适用）

- [ ] ISS-Auth 公钥 / JWT 共享密钥

- [ ] PG 时序分区迁移（`backend/scripts-backend/migrate_partitioned.py --apply`）
- [ ] L1 行业本体导入（manufacturing / energy 内置）
