## 1. 产品定位

### 1.1 一句话定位

**KAP 是面向全行业的、私有化部署的、AI-native 的企业知识治理全流程智能体平台，首批锁定制造与能源。**

### 1.2 三块产品形态

KAP 不是单一产品，而是由三种**产品形态**组成的协同平台。三块共享同一治理底座、同一知识资产、同一用户体系，但服务于不同角色与场景：

| 块 | 名称 | 产品形态 | 主要用户 | 核心价值 |
|---|---|---|---|---|
| **块①** | 知识咨询智能体 | AI 对话式顾问 | DG 数据治理员 + 业务负责人 | 协助客户从无到有建立企业级知识体系（4 级主树 + 多面分类） |
| **块②** | 知识库 + 知识图谱 | 治理工作台 | DG / SME / SEC / AIOps 四角色 | 承载知识内容、实施六工位入库流水线、做持续治理 |
| **块③** | 渐进式消费门户 | 终端用户门户 | 业务一线员工 + 管理层 | Wiki 缓存 + RAG + 图谱推理 三路并行检索，渐进式披露知识 |

### 1.3 行业范围

KAP 定位为**全行业**知识治理平台，通过**可插拔行业模板包**（见 §7.4）适配不同行业的知识体系骨架、本体、命名规范。

- **首批锁定（M0-M5）**：
  - 制造业（含离散制造、装备制造、流程制造）
  - 能源（含电力、油气、新能源、能源装备）
- **第二批扩展（GA 后）**：金融、政企、医疗、运营商、军工
- **第三批扩展（按客户需求）**：交通、教育、科研、零售
- **行业模板共建机制**：客户私有化部署时，平台预置 1 套通用模板 + 1-2 套客户行业模板；如客户行业不在预置范围内，通过块① 知识咨询智能体引导客户共建（参考通用模板演化）
- **不做**：个人 / 小团队级知识管理（钉钉文档、飞书 Wiki 已覆盖）

> **为什么"首批"先做制造能源**：行业标准成熟（ISA-95 / IEC 61850 / ISO 55000 等明确锚点）、Wiki-map V15 已有能源 53 域模板可直接复用、客户付费意愿与决策周期匹配；其他行业的标准与样板可在 GA 后基于平台框架快速扩展。

### 1.4 部署形态

- **唯一形态**：客户私有化部署（On-Premise）
- **不做**：公有 SaaS（行业敏感性决定）
- **必备能力**：单实例 + 多业务单元逻辑隔离、向量与脱敏映射不出域、模型调用经睿动统一网关

### 1.5 目标用户与角色

| 角色代号 | 角色名称 | 主要职责 | 流量占比（预估） |
|---|---|---|---|
| **DG** | 数据治理员（Data Governor） | 分类、元数据、命名规范、版本 | 10% |
| **SME** | 业务专家（Subject Matter Expert） | 内容正确性、本体演化、知识缺口 | 20% |
| **SEC** | 安全审计员（Security & Compliance） | 密级、脱敏、合规 | 5% |
| **AIOps** | AI 运营员（AI Operations） | 模型置信度、提示词调优、效果回归 | 5% |
| **Reader** | 终端业务用户 | 消费知识、提反馈 | 60% |

**边界原则**：
- 块① 的产出（主树 + Facet + L2 本体）是块② 入库的前置依赖
- 块② 的产出（向量库 + 图谱 + Wiki）是块③ 检索的数据源
- 块③ 的反馈（高频问、点踩、知识缺口）回流到块② 的治理工单和块① 的本体演化

## 4. 块① 知识咨询智能体（建体系）

### 4.1 产品形态

**对话式 AI 顾问**，而非给客户一个空白分类工具让其自行画树。

智能体主动引导客户完成以下流程：

```
1. 上传代表性材料（PDF/Word/Excel/...）
        ↓
2. AI 识别行业 + 推断子方向
        ↓
3. AI 提议主树（基于行业模板）
        ↓
4. 业务负责人对话式调整
        ↓
5. AI 提议 Facet 多面标签维度
        ↓
6. AI 提议命名规范
        ↓
7. AI 检测潜在归类冲突 / 重复
        ↓
8. 导出体系方案（含 L2 本体草案）
```

### 4.2 技术依赖

- 行业识别：基于上传材料关键词 + LLM 推断（参考 Wiki-map SkillsRouter 两阶段匹配）
- 主树提议：从平台预置的行业模板包检索 + LLM 微调（行业模板见 §7）
- Facet 提议：基于材料元数据 + 行业 Facet 标准
- 命名规范：模板化规则 + LLM 校验
- 冲突检测：相似度比对 + LLM 一致性检查

### 4.3 知识体系结构（D4）

**双层模型**：主分类（Taxonomy）+ 多面分类（Faceted Classification）

#### 主树 4 级（MECE 约束）

```
Level 1: 公司（或业务单元）
  Level 2: 部门
    Level 3: 岗位
      Level 4: 具体操作 / 工作场景
```

主树承载**唯一归属**，用于权限、Owner、生命周期管理。MECE 在主树上严格成立。

#### Facet 多面（多值，横切）

| Facet 维度 | 取值示例 | 用途 |
|---|---|---|
| 知识类型 | 制度规章 / SOP流程 / 技术标准 / 研发资料 / 故障案例 / 培训教材 / 会议纪要 | 检索过滤 |
| 时效性 | 长期有效 / 周期复审 / 一次性 | 治理（Gardener Agent） |
| 密级 | 公开 / 内部 / 秘密 / 机密 | 召回路由 + 审计 |
| 设备 / 产品维度 | KKS 编码 / 客户自有编码 | 跨部门检索 |
| 标准引用 | GB / IEC / ISO 标准号 | 标准升版联动复审 |
| 业务流程 | 售前 / 研发 / 生产 / 售后 / ... | 流程视图 |
| Owner 角色 | （引用主树第 3 级） | 责任追溯 |

### 4.4 命名规范

每份知识资产命名必须包含：

```
[层级编码]-[业务域代码]-[文档类型]-[标题]-[版本]-[密级]-[Owner]-[生命周期态]
```

示例：`KB-CS-SOP-投诉处理-v2.3-内部-客服部-生效中`

命名规则在体系建设阶段一次性锁定，后期变更需走治理流程。

### 4.5 行业模板（D21）

行业模板是可插拔的标准包（详见 §7），每个包包含：
- L1 主树骨架（参考行业标准）
- L1 Facet 维度
- L1 实体本体（实体类型 + 属性）
- L1 关系本体（关系类型 + 约束）
- Refiner Prompt 模板
- 测试数据集（用于行业模板回归）

## 5. 块② 知识库 + 知识图谱（治理底座）

### 5.1 六工位入库流水线（D5）

```
W1 解析 → W2 自动归类 → W3 切块+元数据 → W4 实体关系抽取 → W5 入库双写 → W6 消费监控
   │           │              │                  │                │           │
   └───────────────────── 工位×角色审核台 ────────────────────────┘     反馈回流
```

#### 工位职责详情

| 工位 | LLM 职责 | 人工职责 | 关键产出 | LLM 主动质疑（Critic）|
|---|---|---|---|---|
| **W1 解析** | 多模态解析（PDF/Word/Excel/PPT/图/音/视）、版式还原、扫描件 OCR | 抽样校验 | 结构化文档对象 | "本文档第 X 页表格识别置信度低" |
| **W2 自动归类** | 主树 + Facet 多标签分类，输出置信度 | 低置信度纠偏 | 分类标签 + 置信度 | "本文与 KB-XXX 相似度 87%，疑似重复入库" |
| **W3 切块+元数据** | 按文档类型自适应切块（SOP / 规章 / FAQ / 技术文档），抽取标题/版本/Owner/标准引用 | 抽样核对 | chunks + 富元数据 | "切块跨越了语义边界" |
| **W4 实体关系抽取** | 基于本体抽实体、关系、属性，含敏感实体标记 | **必审**（核心研发知识） | entities + relations + 敏感标记 | 6 维质疑（见 §5.5） |
| **W5 入库双写** | 向量库 + 图谱同步写入，建立反向索引、相似实体合并提议 | 校验图谱可视化 | Milvus 向量 + Neo4j 图谱 | "新实体与已有 Y 相似度 95%，建议合并" |
| **W6 消费监控** | 召回率/命中率/反馈回流分析、知识缺口识别 | 治理决策 | 仪表盘 + 缺口清单 | "Q 类问题持续低置信，疑似知识缺口" |

#### 切块策略（D5 配套）

按文档类型分流，避免一刀切：

| 文档类型 | 切块策略 |
|---|---|
| 制度 / 规章 | 按章节 + 条目切，保留条款层级 |
| SOP 流程 | 按步骤切，保留前后置依赖关系 |
| FAQ 问答 | 按问答对切 |
| 技术文档 | 按语义段落切 + 图表关联 |
| 会议纪要 | 按议题切 |
| 研发报告 | 按章节 + 公式 / 实验单元保留 |

### 5.2 工位 × 角色矩阵审核台（D6, D7）

#### 4 角色 × 6 工位分工

主审 = R，协审 = C，知会 = I

| 工位 | DG | SME | SEC | AIOps |
|---|---|---|---|---|
| W1 解析 | R | I | C（脱敏标记） | I |
| W2 自动归类 | R | C（跨域） | C（密级） | I |
| W3 切块+元数据 | R | I | I | I |
| W4 实体关系抽取 | I | **R** | C（敏感实体） | I |
| W5 入库双写 | R（合并） | C（冲突） | C（权限映射） | I |
| W6 消费监控 | I | R（知识缺口） | C（泄露排查） | R（命中率） |

#### 审核台数据模型

```
ReviewTask {
  id, batch_id, workstation, object_type, object_id,
  llm_output: {
    content,                # 抽取/分类/切块结果
    confidence,             # 0-1 置信度
    evidence: [...],        # 原文引用句 + 定位
    critique: [...],        # LLM 主动质疑
    proposed_action         # AI 建议决策
  },
  routing: {role, assignee, priority, sla_due_at},
  state: pending | reviewing | approved | rejected | escalated | withdrawn,
  decision: {
    action,                 # approve | modify | reject | merge | split
    modified_content,
    reason,                 # 必填，回流训练用
    timestamp, reviewer
  },
  lineage: {source_doc, parent_task, ontology_version, model_version}
}
```

`reason` 字段必填且结构化（下拉枚举 + 自由文本），用于喂给 LLM 后续提示词优化和模型回归。

#### SLA 超时机制（D12）

**人工审核兜底**：SLA 超时**不允许 LLM 自动通过**。处理策略：
- 超时 → 自动升级到上级专家
- 持续无人审 → 进入"积压告警"，DG 协调资源
- 不存在"LLM 自动通过"路径

W4 工位是 SME 高负载工位（核心研发知识必审），需配套：
- LLM 预填模式（建议决策 + 一键通过）
- 批量审核界面
- 历史相似审核的"判例"复用

### 5.3 双层本体 + 全量重抽（D8, D9）

#### 双层本体设计

```
L1 行业基础本体（平台预置，相对稳定）
  · 制造业基础概念（产品/工艺/物料/设备/工序/缺陷/标准/人员/组织）
  · 能源 IEC CIM 公共信息模型 + KKS 编码体系
  · 平台版本化维护，客户继承

L2 企业扩展本体（客户私有，可演化）
  · 客户专有的产品树、装置编码、专有工艺
  · LLM 在抽取过程中主动提议新增实体类型 / 关系类型
  · 进入审核台，由 SME 审批
```

#### 演化触发条件（LLM 自动监测）

- 同类未匹配实体累计超阈值（如 50 个）→ 提议新实体类型
- SME 手工标注的"自定义关系"反复出现 → 提议固化进本体
- 现有关系类型在不同语境下语义漂移 → 提议拆分
- 行业标准升版（GB / IEC 新版）→ 提议本体扩展

#### 全量重抽机制（核心工程难点）

**影子库切换**：

```
日常入库 → 主图谱（本体 v_n）
                  ↓ 本体变更触发
              影子图谱（本体 v_n+1，全量重抽）
                  ↓ 对比报告 + SME 审批
              灰度切换 → 主图谱
```

重抽期间日常入库**不阻塞**，新数据先进主图谱，本体切换后再补抽。

**增量哈希**：chunk 内容哈希未变 → 跳过实体抽取，仅按新本体重映射；只有 schema 影响的部分才重跑。这是控制成本的命门。

**本体版本化**：每次本体变更打 tag（`ont-v2.3.0`），所有实体 / 关系挂版本号；图谱查询默认走 latest，但支持 `as_of` 历史回溯（审计需求）。

**触发频率**：
- 默认季度自动触发
- **客户可配置**自动触发频率（按月 / 按季 / 按半年 / 按变更事件）
- 也支持手动触发

**回滚机制**：新本体上线后 7 天观察期，任何指标恶化（召回率、SME 驳回率、命中率）可一键回退到 `ont-v(n-1)`。

### 5.4 敏感实体识别与脱敏（D10, D11）

#### 三类敏感数据各自方案

| 类别 | 识别方式 | 脱敏策略 | 一致性关键 |
|---|---|---|---|
| **人名** | NER + 客户白名单 + 角色字典 | **角色化替换**："张工" → "研发员A" | 全局映射表保证跨文档 ID 一致，否则关系图谱断链 |
| **工艺参数数值** | 正则 + 单位识别 + 本体属性绑定 | **三级降精度**：精确值 / 区间值 / 等级标签 | 区间宽度由 SME 按知识价值定，不能一刀切 |
| **客户名** | 实体识别 + 客户主数据匹配 | **代码化**："客户A001"，按角色解码 | 与客户主数据系统对接，避免重复编码 |

#### 双层存储 + 双向量

```
原文（密文存储，按密级 KMS 加密）
    ↓
脱敏文（明文存储）
    ↓
向量化两份：
  · vec_redacted（脱敏向量，默认召回路径）
  · vec_original（原文向量，仅高密级用户访问）
    ↓
查询时：按用户密级路由召回 → 展示时按密级解码映射
```

#### 工位嵌入

- W1 解析后 → 脱敏 Agent 标记敏感片段（不修改原文，仅打 span 标签）
- W4 实体抽取 → 敏感实体打 `is_sensitive` 标签 + 关联映射 ID
- W5 入库 → 双层存储 + 双向量
- W6 消费 → 按用户密级实时路由 + **每次原文访问写审计日志**

#### 脱敏映射表存储（D11）

**加密 KV** 存储（不上 HSM）：
- 选型理由：制造能源企业绝大多数客户密级到"秘密"级别为止，加密 KV 已经够用；HSM 成本高，仅在客户明确要求时（涉密资质企业）启用
- 实现：Redis + 应用层 AES-256 加密 + 密钥独立管理（KMS）
- 访问审计：映射表的每次读取都写审计日志

### 5.5 LLM-Critic 主动质疑机制（D13）

#### 核心模式：双 Agent 互审

```
抽取 Agent（LLM-A）出结果
        ↓
质疑 Agent（LLM-B，独立提示词）找问题
        ↓
审核台（统一界面）
```

让第二个 LLM 用不同提示词专门"挑刺"，输出结构化质疑清单。

#### 6 维质疑清单

| 维度 | 触发条件 | 示例 |
|---|---|---|
| **一致性** | 实体定义在不同文档中冲突 / 关系在不同来源中矛盾 | "实体 [振动分析工程师] 在 KB-DEV-0091 中定义为 [设备诊断岗]，疑似同义" |
| **完整性** | 本体要求该实体应有属性 X，但文档未提及 | "实体 [汽轮机1#] 缺少必填属性 [位号KKS]" |
| **证据强度** | 关系仅有一句模糊表述支撑 | "关系 [执行] → [GB/T 6075] 证据强度低，仅一句模糊表述" |
| **重复性** | 新实体与图谱已有节点高度相似（向量+字面双判） | "新实体 [汽轮机一号] 与 [汽轮机1#] 相似度 95%，建议合并" |
| **时效性** | 引用的标准 / 制度版本已作废 | "引用的 GB/T 6075-2003 已于 2012 年作废，现行版本为 GB/T 6075-2012" |
| **跨域** | 跨文档的关联推断（最有价值） | "本文工艺参数范围与同类设备的历史故障案例存在关联" |

#### LLM-Critic 与 Wiki-map 治理 5 Agent 的关系

KAP 的 LLM-Critic 是 Wiki-map V15 治理 5 Agent 的**扩展与升级**：

| Wiki-map V15 | KAP 升级 |
|---|---|
| Auditor（溯源审计） | 升级为 6 维质疑中的**证据强度 + 完整性** |
| Standardizer（实体归一化） | 升级为**重复性**质疑 |
| Gardener（冷门淘汰） | 升级为**时效性**质疑 |
| Curator（缺口起草，stub） | 升级为 W6 的**知识缺口识别** |
| Deduper（去重，stub） | 升级为**一致性 + 重复性** |
| 无 | 新增**跨域**质疑（最有价值） |

### 5.6 专家共识机制（D14）

当多名 SME 对同一抽取结果意见分歧时：

```
分歧检测（≥2 名 SME 决策不一致）
        ↓
LLM 推荐合并方案（基于双方理由 + 历史判例）
        ↓
专家委员会投票（3-5 名 SME，少数服从多数）
        ↓
未达共识 → 升级到首席业务专家
        ↓
达成共识 → 写入"判例库"，供后续相似争议参考
```

判例库逐渐沉淀为客户的"业务知识仲裁规则"，是 SaaS 化的反例——这恰恰是私有化的核心价值。

### 5.7 知识图谱设计

#### 实体类型（L1 行业基础）

| 类型 | 说明 | 关键属性 |
|---|---|---|
| 组织 | 公司 / 部门 / 项目组 | 编码、层级、Owner |
| 人员角色 | **角色而非个体**（配合脱敏） | 岗位、权限域 |
| 产品 / 装置 | 带 KKS 或客户编码 | 编码、版本、生命周期态 |
| 部件 / 物料 | BOM 节点 | 物料号、规格、供应商域 |
| 工艺 / 工序 | 制造流程节点 | 工序号、参数集、标准引用 |
| 设备 | 资产实体 | 资产号、位号、所属装置 |
| 标准 / 规范 | GB / IEC / ISO 等 | 标准号、版本、生效日 |
| 制度 / SOP | 内部规章 | 文号、版本、Owner、复审日 |
| 故障 / 缺陷 | 质量与运维核心 | 编码、严重度、处置 |
| 试验 / 检验 | 研发与质检 | 试验号、方法、结论 |
| 参数指标 | 工艺与设备参数 | 名称、单位、阈值范围 |
| 文档 | 元节点（挂所有证据） | 文档 ID、版本、密级 |

#### 关系类型（L1 行业基础）

- `组成`：产品 → 部件、装置 → 设备
- `执行`：人员角色 → SOP、工序 → 工艺
- `引用`：SOP → 标准、制度 → 法规
- `影响`：参数 → 质量、参数 → 故障概率
- `因果`：故障 → 缺陷、检验 → 处置
- `替代 / 演化`：部件 v2 → 部件 v1、标准新版 → 旧版
- `适用`：标准 → 产品系列、SOP → 设备类
- `证据于`：所有节点 / 关系 → 文档（**最重要，溯源根基**）

#### 双视角图谱（沿用 Wiki-map V15）

- **实体视角**：节点 = 实体，边 = 实体间关系
- **文档视角**：节点 = 文档，边 = 共享实体的关联关系（用于发现知识聚类与孤岛）

#### 持久化（D8 配套）

- 主图谱：Neo4j（持久化，避免 Wiki-map V15 内存模式重启丢失的踩坑）
- 影子图谱：Neo4j 独立实例 / 数据库（全量重抽专用）
- 双向索引：`_nodes`（实体 → 文档）+ `_edge_index`（去重）+ `_synonym_map`（归一化）

## 6. 块③ 渐进式消费门户

### 6.1 三路并行召回 + 重排序融合（D15）

```
查询
  ↓
[Wiki 精确匹配 + 向量召回 + 图谱推理] 三路并行同时打分
  ↓
重排序（融合三路 + 用户密级过滤）
  ↓
[Wiki 高置信度 ≥ 0.90 → 短路返回]
  ↓
[融合答案 + 证据链 + 路由徽章]
```

**与 Wiki-map V15 的区别**：
- V15 是 IntentRouter → SkillsRouter → 单路检索的**串行路径**
- KAP 是**召回阶段并行**，重排序阶段融合，避免"Wiki 有相似但不准的问答 → 直接返回错答"

#### 三路打分

| 路径 | 打分维度 | 权重 |
|---|---|---|
| Wiki 缓存 | 标题精确匹配 + 内容相似度 | 高（短路用） |
| RAG（向量） | 向量相似度 + BM25 | 中 |
| 图谱推理 | 实体路径长度 + 路径置信度 | 中 |

最终 HybridScorer：`α·向量 + β·图谱 + γ·目录(Wiki) + δ·BM25`，权重可配置（继承 Wiki-map V15 的实现）。

### 6.2 Wiki 缓存机制（Karpathy 三层）

继承 Wiki-map V15 的 Karpathy LLM Wiki 模式：

- **Raw 层**：原始文档不可变，作为最终事实源
- **Wiki 层**：LLM 编译产出的高频 / 标准答案，Markdown 格式，支持 `[[link]]` 跨页跳转 + `[← doc]` 溯源
- **三层索引**：domain（一级域）/ category（细分）/ topic（具体页）

### 6.3 频率回流闭环（D16）

```
高频问候选（W6 监控产出）
        ↓
LLM 自评（答案是否稳定且正确？置信度？）
        ↓ 置信度 ≥ 阈值
SME 抽审（人工质量门）
        ↓ 通过
入 Wiki（version+1，挂 source_doc_ids）
        ↓
后续命中走 Wiki 短路
```

**关键质量门**：频率高 ≠ 答案对。若直接 promote，Wiki 会变成"错误答案的固化器"，越用越烂。

### 6.4 反馈驱动的知识缺口

```
用户点踩 + 低置信度回答 + 无答案查询
        ↓
W6 知识缺口识别（聚类 + 频率）
        ↓
缺口工单 → SME 审核台
        ↓
SME 决策：
  · 起草 Wiki（Curator Agent 协助）
  · 录入新文档
  · 标记"暂无标准答案，转人工"
```

形成完整的 PDCA 闭环，区别于"普通 RAG demo"的关键就在这里。

### 6.5 双门户对应

| 门户 | 路径 | 主要用户 |
|---|---|---|
| 消费门户（Reader） | `/portal/read` | 终端业务用户 |
| 治理工作台（Editor） | `/workbench/...` | DG / SME / SEC / AIOps |

参考 Wiki-map V15 的 EditionPill 软切换机制：同上下文不丢项目，模式间编辑完即时同步。

## 8. 安全与合规

### 8.1 RBAC + 数据权限 + 密级三维

#### 4 角色 RBAC（D7）

继承 ISS-Auth 的 JWT + Redis 会话方案：
- DG / SME / SEC / AIOps / Reader 五角色
- `@RequiresPermissions` 注解 + `@RequiresRoles` 注解（AOP 拦截）
- `@InnerAuth` 用于服务间内部调用

#### 5 级数据权限（继承 ISS-Common-Datascope）

| 级别 | 说明 |
|---|---|
| 1 | 全部数据（公司级管理者） |
| 2 | 自定义部门集合（跨部门协作） |
| 3 | 仅本部门 |
| 4 | 本部门及子部门（基于 ancestors 链路 + FIND_IN_SET） |
| 5 | 仅本人 |

多角色 OR 取并集；防 SQL 注入兜底；超级管理员跳过。

#### 密级（KAP 新增维度）

```
公开 → 内部 → 秘密 → 机密
```

密级与数据权限**正交**：用户首先满足数据权限，其次满足密级匹配，才能访问。

#### 召回阶段过滤（关键）

密级过滤必须在**召回阶段**就执行（而非展示阶段），即 Milvus 查询的 `where` 子句直接带 `access_level <= user_level`，避免高密向量被低密用户的查询命中后再过滤——既低效又有泄露风险。

### 8.2 全链路审计

继承 ISS-Common-Log + OpenTelemetry：
- 所有 LLM 决策可追溯：模型版本 + Prompt 版本 + 证据 + 用户 + 时间
- 所有原文访问写审计日志
- 所有审核台决策保留 reason + 时间戳 + 审核人
- 全链路 trace_id 串联前端 → 网关 → 业务 → AI → LLM → 存储

### 8.3 私有化部署强约束

| 约束 | 说明 |
|---|---|
| 向量不出域 | 嵌入模型必须本地化（bge-large-zh / bce-embedding） |
| 脱敏映射不出域 | 加密 KV 在客户内网 |
| 模型经睿动网关 | 客户可配置本地模型 / 国产模型 |
| 不硬编码模型名 | 通过 `/v1/models` 动态拉取 + 客户端过滤 |
| API Key 后端持有 | 前端通过 `/api/v1` 代理，禁止前端直连模型 |
| SSO 密钥不入代码 | AES_KEY / AES_IV 走配置中心或环境变量 |

## 12. 关键数据流（穿透链）

### 12.1 文档入库穿透链

```
客户上传文档（块② 治理工作台 / 集成 API）
   ↓
ISS-File 多对象存储
   ↓ MQ 异步任务
ISS-Knowledge-Parser（W1 解析）
   ↓ AsyncClient 调睿动多模态
   ↓ 输出结构化文档对象
KAP-Knowledge-Core
   ├─ W2 SkillsRouter 自动归类 → 审核台（DG 主审）
   ├─ W3 切块 + 元数据
   ├─ W4 LLM-A 抽取实体关系 → LLM-B 6 维质疑 → 审核台（SME 主审）
   ├─ 敏感实体识别 + 脱敏（W4 嵌入）
   └─ W5 双层存储双向量入 Milvus + 持久化入 Neo4j
   ↓
W6 监控统计入仪表盘
```

### 12.2 查询消费穿透链

```
用户在块③ 输入查询
   ↓
密级路由：用户密级 → 召回过滤条件
   ↓
[Wiki 精确匹配 + 向量召回 + 图谱推理] 三路并行
   ↓
HybridScorer 重排序
   ↓ 若 Wiki 命中度 ≥ 0.90 → 短路返回
   ↓ 否则
LLM 融合答案 + 证据链 + 路由徽章
   ↓
答案 + 溯源（[← doc] 弹原文）+ 反馈按钮
   ↓
用户反馈（点踩 / 点赞）→ W6 知识缺口识别
```

### 12.3 治理工单穿透链

```
LLM-Critic 6 维质疑产工单 → GovernanceQueue
   ↓
按工位 × 角色路由分发到对应审核台
   ↓
SME / DG / SEC / AIOps 在审核台决策
   ↓ 决策类型：approve / modify / reject / merge / split
   ↓
工单状态机变更 → 知识库更新（version+1）
   ↓
决策 reason 回流 → LLM 提示词优化数据集
```

### 12.4 本体演化穿透链

```
LLM-Critic 监测到本体演化触发条件
   ↓
LLM 提议新增 / 拆分 / 合并实体 / 关系类型
   ↓ 进入审核台（SME 主审）
   ↓ 通过
本体 v_n+1 入库 + 影子图谱启动全量重抽
   ↓
对比报告（新旧本体的实体覆盖率、关系数量、检索命中率）
   ↓ SME 审批 + 灰度切换
主图谱切到 v_n+1
   ↓ 7 天观察期
若指标恶化 → 一键回退到 v_n
```

### 12.5 编辑回流穿透链

```
路径 A（治理侧）：审核台工单点 [改]
路径 B（消费侧）：Wiki 阅读时点 [编辑]
   ↓ 同一组件
WikiEditorModal
   ↓ PUT /api/v1/wiki/pages/{id}（upsert）
WikiStore version 自动 +1
   ↓ 路径 A → 工单 decide('edit') → 状态 edited
   ↓ 路径 B → WikiReader 自动刷新显示新版
```

## 13. 已踩过的坑（来自 Wiki-map V15 实测）

| 坑 | 影响 | KAP 解决方案 |
|---|---|---|
| 同步 `httpx.Client` 阻塞 uvicorn 事件循环 | ingest 期间所有 API 挂起 | **必须用 AsyncClient**，所有 LLM 调用全异步 |
| Milvus 不稳定 / 连接断开重连超时 | 向量检索失败 | 连接池 + 健康检查 + 自动重连（参考 ISS-Knowledge-Parser 的 ConnectionManager） |
| Judge 阈值过严，KEEP 率仅 42% | 大量文档误归档 | 阈值可配置，按行业模板预设不同阈值 |
| domain_id 分配不精，全归 `regulation` 通用域 | 分支检索失效 | **必须给行业模板加 Refiner Prompt 指引**，模板中含具体 domain 关键词字典 |
| 图谱内存模式重启丢失 | 服务重启需重灌 | **Neo4j 持久化** |
| Mock embedding 无语义区分 | 召回精度低 | 私有化必须接入真嵌入（bge-large-zh / bce-embedding） |
| 单角色 Editor 无法满足企业 RBAC | 治理无法分工 | 4 角色矩阵化 + ISS-Common-Datascope |
| 仅 access_level 二元过滤 | 密级不分级 | 4 级密级 + 召回阶段过滤 |

## 14. 实施路线建议

### 14.1 阶段划分

| 阶段 | 重点 | 预估周期 |
|---|---|---|
| **Phase 0**：基础准备 | 私有化基建（K8s + Nacos + 存储）+ ISS 底座部署 + 睿动接入 | 2 周 |
| **Phase 1**：块② MVP | W1-W5 工位 + 4 角色审核台 + 制造 / 能源 L1 模板 + 敏感脱敏 | 8 周 |
| **Phase 2**：块③ MVP | 三路并行召回 + Wiki 缓存 + 反馈回流 | 4 周 |
| **Phase 3**：块① MVP | 知识咨询智能体（对话式建体系） | 4 周 |
| **Phase 4**：本体演化 | 双层本体 + 全量重抽影子库 + 回滚 | 4 周 |
| **Phase 5**：精品打磨 | 专家共识 + 判例库 + 6 维质疑全量上线 + UI 重设计 | 6 周 |
| **Phase 6**：客户 PoC | 1-2 家制造 + 1-2 家能源客户的真实数据验证 | 8 周 |

**总计**：约 36 周（9 个月）至 GA

### 14.2 里程碑

- M1（Phase 1 末）：能跑通 W1-W5 + 4 角色审核 + 制造行业 L1 模板
- M2（Phase 2 末）：消费门户能跑通三路并行召回，命中率 ≥ 60%
- M3（Phase 3 末）：知识咨询智能体能引导客户从 0 建立 4 级体系
- M4（Phase 4 末）：本体演化与全量重抽闭环跑通
- M5（Phase 6 末）：首家客户 PoC 验证通过

### 14.3 关键依赖

- 睿动网关稳定性 + 模型选型（生产推荐 glm-5.1 / Qwen-Max / Claude Sonnet）
- 嵌入模型选型与微调（bge-large-zh 通用 / 行业微调可选）
- 客户 PoC 数据（真实文档 + 真实查询日志，至少 1 万份文档）
- SME 资源（核心研发知识必审，需客户配合提供专家时间）

### 15.1 已识别风险

| 风险 | 等级 | 缓解策略 |
|---|---|---|

| LLM 幻觉导致质疑误报 | 中 | LLM-Critic 自身也需置信度阈值 + SME 反馈调优 |
| Java + Python 混合架构复杂度 | 中 | 服务边界清晰 + Feign 调用 + OTEL 追踪 |
