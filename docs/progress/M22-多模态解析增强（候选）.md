# M22 多模态解析增强 + 知识中心补强（候选清单）

> 本期分两条主线：**A 主线 = 借鉴 RAG-Anything 的多模态解析增强**（4 commit）+ **B 补强 = KAP 知识中心自驱薄弱项补齐**（3 commit）。两条线独立可分批落地，也可单独裁掉。

## 立项依据

**外部参考**：[HKUDS/RAG-Anything](https://github.com/HKUDS/RAG-Anything)（基于 LightRAG，All-in-One 多模态 RAG 框架）

**KAP 当前痛点**（知识中心 W2/W6 链路）：
- `backend/packages/storage/parsers/pdf_parser.py` 用 pdfplumber，**只抽纯文本**，丢失版式 / 表格行列关系 / 公式
- `parsers/` 下无独立 **table_parser / equation_parser**，表格被 chunker 切成纯文本失去逻辑关系
- `chunker.py` 不做 **context-aware** 注入，图/表/公式被孤立处理
- `extraction/entity_extractor.py` 单文件、纯文本路径，不进图像与表格

**业务驱动**：制造业图纸 / 能源工程规范 / 化工 SOP / 装备 BOM 这类文档天然多模态，当前链路丢的信息会直接影响：
- W4 抽取诊断分数（M20 已埋点）
- W5 Wiki 编织质量（M17-M19 6 维评分）
- 召回率 / 精确率（M8-M10 评估体系）

**为什么不照搬 RAG-Anything 全栈**：见 [借鉴判断对照表](#与-rag-anything-差异化对照)，KAP 的 4×6 矩阵审核台 / 双层本体演化 / 影子库重抽 / 自学习闭环不能被替代。

## A 主线 · 多模态解析增强 commits（4 个）

### M22 #1 · MinerU 接入 W2 PDF 解析链（高保真）

**落点**：
- `backend/packages/storage/parsers/pdf_parser.py` 加 **多后端切换**
  - 新增 `MinerUBackend`（保留版式 / 公式 LaTeX / 表格结构）
  - 保留 `PdfplumberBackend`（轻量回退）
  - 保留 `MockBackend`（无 dep 时回退，沿用现有 `_parse_mock`）
- 环境变量：`KAP_PDF_PARSER=mineru|pdfplumber|mock`（默认 `pdfplumber`，dev/prod 可切 mineru）
- `ParsedContent` 扩展字段：
  - `tables: list[TableBlock]`（行列 + 单元格语义）
  - `equations: list[EquationBlock]`（LaTeX + 周边上下文）
  - `images: list[ImageBlock]`（base64 / minio 引用 + bbox + caption）
- `pyproject.toml` 加 optional dep group `[mineru]`，避免基础镜像膨胀

**资源评估**：
- MinerU 单页 ~200-500ms（CPU），GPU 加速 5-10x
- 私有化部署需评估增量：建议方案 A 的 `kap-storage` 容器（已 4 vCPU/8 GB）增 2 GB 内存
- mineru 模型 ~3 GB，加入镜像或挂载 volume

**测试**：
- +3 backend（MinerU mock backend / pdfplumber 不变 / 切换开关）
- +1 live_llm（真 PDF 跑 mineru，标 `@pytest.mark.live_llm`，CI 默认 skip）

**对接**：W4 抽取诊断 `extraction_quality.py` 自动统计 `table_count / equation_count`，进 M20 趋势

---

### M22 #2 · 表格 / 公式专用处理器

**落点**：
- `backend/packages/storage/parsers/table_parser.py`（新）
  - 输入 `TableBlock`（来自 MinerU 或外部），输出 **行列语义 chunk**
  - 策略：表头识别 → 每行作为独立 chunk + 表头注入 metadata.facet
  - 大表（>50 行）滚动窗口切片，保留上下表头
- `backend/packages/storage/parsers/equation_parser.py`（新）
  - LaTeX 保留 + 周边文本上下文（前后 200 字）一同入 chunk
  - chunk metadata 加 `content_type=equation`，便于 W5 Wiki 编织时单独处理
- `parsers/router.py` 改：
  - 解析后产物按 content_type 路由（text / table / equation / image）
  - 各自走对应 chunker 策略

**测试**：+5（table 行级切片 / 表头注入 / 公式 LaTeX 保留 / context 注入 / router 多 type 分流）

---

### M22 #3 · Context-Aware Chunking（轻量补丁）

**落点**：
- `backend/packages/storage/chunker.py` 加 `enrich_with_context` 选项
  - 处理 image / table / equation chunk 时，把前后 N 个 chunk 的文本拼到 `metadata.context_window`
  - N 由 `settings.context_window_size` 控制（默认 2）
- `backend/packages/extraction/entity_extractor.py` 改：
  - 抽实体时如有 `context_window`，prompt 里一并喂 LLM
  - 弱断言测试 + 真 LLM 跑（M13 #1 真 LLM 测试基建已就绪）

**为什么轻量**：约 50 行代码 + prompt 模板调整，但对 W4 抽取诊断分数提升预期最高（图/表上下文常是知识图谱缺失边的来源）

**测试**：+3（context 注入 / window size 配置 / 实体抽取 prompt 带 context）

---

### M22 #4 · ISS 解析结果 bypass 入口（参考 RAG-Anything Direct Content List Insertion）

**落点**：
- `backend/api/routers/knowledge.py` 加 `POST /api/v1/knowledge/structured-chunks`
  - 接收已结构化的 chunks（外部 ISS-Knowledge-Parser 或客户既有解析器送来）
  - schema：`{doc_id, project_id, chunks: [{content, content_type, metadata}], parser_name}`
  - 跳过 parsers/，直接进 chunker 后置流程（embedding + storage + 实体抽取 + 入图）
- 鉴权：`scope=knowledge:write`
- 审计：审计日志记录 `parser_name + source_system`，便于追溯

**业务场景**：
- 客户已用商用 OCR / 解析平台（如 ABBYY、TextIn），不希望重复解析
- ISS-Knowledge-Parser 既有产物直接复用
- 单元测试 / 调试时跳过解析直接喂语料

**测试**：+4（schema 校验 / 鉴权 / 入库链路完整 / 审计落库）

## B 补强 · 知识中心薄弱项补齐 commits（3 个）

> 与 RAG-Anything 借鉴无直接关系，是 KAP 知识中心自驱薄弱项。引入 MinerU 后，extraction 压力 + W6 吞吐压力 + 重抽成本都会上升，**A 主线落地后这 3 项几乎是必要前置**。

### M22 #5 · 关系抽取器 + 实体消歧（补强 `extraction` 包）

**痛点**：
- `backend/packages/extraction/entity_extractor.py` **单文件**，只抽实体，**不抽关系**
- 没有实体消歧（同一实体不同表述会变成多个图谱节点）
- M22 #2 表格 / 公式入图后实体数量预期翻倍，关系抽取缺失会让图谱碎片化

**落点**：
- 新增 `backend/packages/extraction/relation_extractor.py`
  - 输入：chunk + 已抽实体 → 输出关系三元组（with confidence）
  - LLM 提议 → 进 4×6 矩阵审核台（**复用** M0+M3 governance，不绕过 D12）
  - 关系类型从 L2 本体读取（关系也是双层本体的一部分）
- 新增 `backend/packages/extraction/entity_resolver.py`
  - 候选合并：embedding 相似度（>0.92）+ 表面字符串规则（编辑距离）
  - LLM 仲裁高歧义对，结果进决策日志（M6+M7）
  - 不自动合并 type=L1 实体（行业本体不变）
- `entity_extractor.py` 改造：抽完实体后调 `entity_resolver.resolve_candidates`
- API：`POST /api/v1/governance/entity-merge-decision`（SME 审核合并提议）

**测试**：+8（relation 抽取 3 / resolver 候选 3 / LLM 仲裁 1 / 决策日志落库 1）+ 1 live_llm

**工作量估**：~4h

### M22 #6 · W6 入库可观测性补齐（throughput / latency / failure）

**痛点**：
- M7+M8 已埋查询召回 metrics、M19+M20 已埋 Wiki / W4 抽取诊断
- **W6 入库本身没埋点**：吞吐 / 单文档延迟 / 失败率 / 阶段耗时（parse / chunk / embed / write）都没数
- MinerU 接入后单页解析时延会变大，没监控会盲飞

**落点**：
- 新增 `backend/packages/observability/ingest_metrics.py`
  - 阶段耗时：`parse_ms` / `chunk_ms` / `embed_ms` / `vector_write_ms` / `graph_write_ms`
  - 计数：`doc_total` / `doc_success` / `doc_failed` / `chunk_total`
  - 错误分桶：`parse_error` / `embed_timeout` / `vector_full` / `graph_violation`
- PG 表 `ingest_metrics`（仿 `extraction_metrics` 模式，M20 #1 模板）
  - 启动水化最近 N 条、INSERT-only write-through、`KAP_INGEST_METRICS_PG=1` 开关
- API：`GET /observability/ingest-metrics/trend`（小时桶 / 天桶）
- 仪表盘 hook：M10 综合仪表盘 API 加 `ingest_throughput / ingest_p95_latency` 字段

**测试**：+6（5 阶段耗时埋点 / 错误分桶 / PG sink / 趋势 API / 仪表盘字段 / 告警阈值）

**工作量估**：~3h

### M22 #7 · 增量重抽 lite 模式（影子库 partial-rebuild）

**痛点**：
- 现状（M4）：影子库 **全量重抽**，按季度跑，资源密集
- MinerU 接入后单文档解析成本 ↑10x，全量重抽对中型客户（10万+ 文档）成本不可接受
- 大部分实际场景：本体演化只动了少数关系类型 → 全量重抽 99% 是浪费

**落点**：
- `backend/packages/rebuild/` 加 `incremental.py`
  - 输入：本体 diff（哪些 type / relation 变了，来自 L2 演化日志）
  - 影响面分析：扫 `entity_index` 找 type 命中的文档集合
  - 只重抽命中文档（chunk + 实体 + 关系），其他文档 chunk 复用
  - 输出 `RebuildPlan{full_docs, partial_docs, skipped_docs, est_cost}`
- API：`POST /api/v1/rebuild/incremental {ontology_version_from, ontology_version_to}`
- 7 天观察期 + 灰度切换 **保持不变**（D8 锁定）
- 全量重抽路径保留（兼容旧路径 + 大版本升级场景）
- CLI：`backend/scripts-backend/rebuild_incremental.py --from v1.0 --to v1.1 --dry-run`

**测试**：+7（diff 解析 / 影响面分析 3 场景 / RebuildPlan 生成 2 / 干跑模式 / 灰度兼容）

**工作量估**：~3h

## 与 RAG-Anything 差异化对照

| RAG-Anything 模块 | KAP 借鉴方式 | 不照搬原因 |
|:---|:---|:---|
| MinerU 高保真解析 | **直接接入**（M22 #1） | — |
| Image/Table/Equation 处理器 | **接入 table/equation**（M22 #2），Image 留 M23+ | 图像语义理解依赖 VLM，资源成本评估后再做 |
| Context-Aware Processing | **直接借鉴**（M22 #3） | — |
| Direct Content List Insertion | **接入**（M22 #4） | — |
| LLM 自动 KG 构建 | **不照搬** | 违反 D12 人工兜底；KAP 走 LLM 提议 + 4×6 矩阵审核 + 全量重抽 |
| LightRAG 4 类 pluggable storage 抽象 | **不照搬** | KAP `storage/` 14 个业务 store 粒度更细，重构成本 > 收益 |
| Hybrid Retrieval（图 + 向量加权） | **不照搬** | KAP 已有 Wiki/RAG/图谱三路并行召回（D15），M0+M2 完成 |
| 跨模态实体进 KG | **延后**（M23+ 候选） | 需先评估对 4×6 矩阵审核台工作量的影响 |

## 风险与依赖

- **MinerU 模型授权**：Apache 2.0，可商用，私有化部署无授权风险
- **MinerU 资源成本**：CPU 单页 ~200-500ms，能源/制造业 100+ 页技术文档需评估批处理排队（建议进 W2 异步队列，复用 M11 ChunkHashCache 模式）
- **依赖版本**：`mineru>=2.0` 与现有 pdfplumber 共存无冲突
- **对 W4 抽取诊断的影响**：表格 / 公式入图后，`extraction_metrics` 字段需扩（M20 #1 17 字段 → 预计 21 字段），需 PG 迁移脚本
- **前端无改动**：W2 上传 UI / W4 诊断看板自动消费新字段，无需 frontend PR

## 验收口径

**A 主线**：
- ✓ MinerU 后端能解析带表格 / 公式的 PDF，产物含独立 `tables/equations` 字段
