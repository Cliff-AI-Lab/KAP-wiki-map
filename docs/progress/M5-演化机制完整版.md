## 全景成果

### 监测条件 2 · 自定义关系反复出现 → 提议固化（M5 #1）
SME 在审核台手工标注的"自定义关系"频次超阈值（默认 20）→ LLM 归纳为新关系类型 → OntologyEvolutionProposal（含 OntologyRelationType）。源 / 目标实体类型由 LLM 推断（约束在已注册 entity_types）。
- **关联代码**：[`packages/ontology/evolution_proposer.py`](../../backend/packages/ontology/evolution_proposer.py) · `propose_relation_solidification`
- **prompt**：[`packages/ontology/prompts.py`](../../backend/packages/ontology/prompts.py) · `RELATION_SOLIDIFY_SYSTEM/USER`

### 监测条件 3 · 关系语义漂移 → 提议拆分（M5 #1）
某关系（如 `governs`）在不同语境下被用于多种语义（标准约束 vs 行政规范）→ LLM 聚类判断 → 拆分为 ≥2 个细化关系。返回 list[Proposal]，should_split=false 时返回 None。
- **关联代码**：[`packages/ontology/evolution_proposer.py`](../../backend/packages/ontology/evolution_proposer.py) · `propose_relation_split_for_drift`
- **prompt**：`RELATION_SPLIT_SYSTEM/USER`

### 监测条件 4 · 行业标准升版 → 提议本体扩展（M5 #1）
客户文档引用了新版标准（如 GB/T 6075-2024 替代 -2012）→ LLM 比对 L1 standard 实体类型 examples → 生成升版后 examples（旧版标 [作废] 保留以便溯源）。
- **关联代码**：[`packages/ontology/evolution_proposer.py`](../../backend/packages/ontology/evolution_proposer.py) · `propose_standard_upgrade`
- **prompt**：`STANDARD_UPGRADE_SYSTEM/USER`

### 7 天观察期 + 指标采集（M5 #2 · 决策书 §5.3 promote 后保护）
promote 成功 → 立即记录 baseline 快照 → 7 天内 SME / 定时器调 `tick_observation` 采样对比基线 → 漂移超阈值 → status=alert（不自动 rollback，让 SME 决策）。
- **告警阈值**：实体数变化 > 50% / 关系数变化 > 50% / 自定义关系比例增长 > 200% / 关键类型（baseline 占比 ≥ 10%）消失
- **接线**：`promote_shadow` 调 `start_observation`；`rollback_promotion` 调 `mark_rolled_back`
- **关联代码**：[`packages/rebuild/promotion_observer.py`](../../backend/packages/rebuild/promotion_observer.py) + [`metrics_collector.py`](../../backend/packages/rebuild/metrics_collector.py)

### 观察期 API endpoints（3 新增）
- `GET  /api/v1/rebuild/observations`              列出（query: project_id 可选）
- `GET  /api/v1/rebuild/observations/current`      当前活跃（404 = 无）
- `POST /api/v1/rebuild/observations/tick`         手动触发采样 + 检查（SME）
- **关联代码**：[`api/routers/rebuild.py`](../../backend/api/routers/rebuild.py)

### ChunkHashCache PG 持久化（M5 #3 · 决策书 §5.3 跨周期复用）
M4 lite 内存缓存仅当次 rebuild job 有效；M5 加 `PgChunkHashCache`：启动 initialize 建表 + 全量加载到内存（write-through），rebuild 完成后 `flush()` 异步批量 upsert。
- 工厂 `get_chunk_hash_cache()`：环境变量 `KAP_CHUNK_HASH_PG=1` 时返回 PG，PG 连不上 → 静默降级内存模式
- **接线**：`arun_rebuild` 完成时检测 `cache.flush` 存在 → await 一次落 PG
- **DDL**：`chunk_hashes(chunk_id PK, content_hash, updated_at)`
- **关联代码**：[`packages/rebuild/incremental_hash.py`](../../backend/packages/rebuild/incremental_hash.py)
