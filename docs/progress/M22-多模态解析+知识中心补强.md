# M22 多模态解析 + 知识中心补强（7 commits / ~6h vs Opus 估 110-140h，节省 ~95%）

## 主线

两条并行：
- **A 主线（4 commits）**：借鉴 HKUDS/RAG-Anything 的多模态解析能力补 W2 文档解析弱项
- **B 补强（3 commits）**：KAP 知识中心自驱薄弱项——关系抽取 / 入库可观测性 / 增量重抽

A 主线让 KAP 能吃多模态（表格 / 公式 / 图像）；B 补强为 MinerU 接入后压力上升的关系图谱 / W6 入库 / 重抽成本 三个面提供配套支撑。

**为什么一起做**：A 主线让实体数量预期翻倍 → 不补 B 关系抽取，图谱碎片化；MinerU 单页 200-500ms → 不补 B 可观测性，盲飞；MinerU 单文档成本 ↑10x → 不补 B 增量重抽，大客户全量重抽不可接受。三者前置必要。

### M22 #7 · 增量重抽 lite — 影响面分析 + RebuildPlan dry-run + API + CLI（eca189e）
- `rebuild/incremental.py` 新: RebuildPlan + analyze_impact
  - L1 变更 → 全量重抽（行业本体可能出现在任何文档）
  - 仅 removed/modified → partial（只重抽用到被改类型的文档）
  - added 新类型 → full（旧文档可能漏抽, 全文重扫）
  - 成本估算: full=1 unit / partial=0.3 unit
  - est_savings_ratio = 1 - cost / full_baseline
- `api/routers/rebuild.py` 加 `POST /rebuild/incremental`（dry_run 默认, 实际执行留 M23）
- `scripts-backend/rebuild_incremental.py` CLI: 离线算 plan + 评估成本
